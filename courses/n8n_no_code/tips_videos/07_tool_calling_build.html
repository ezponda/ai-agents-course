0 · De qué va esta clase y este workflow

~5 min
🗣️ Abrir la clase (lee esto)

"En las clases anteriores montamos nuestro primer AI Agent: un LLM que tiene memoria y puede usar herramientas como la calculadora o Wikipedia. Vimos que el agente decide cuándo usar cada herramienta. Hoy abrimos esa caja negra y respondemos a la pregunta clave: ¿cómo decide? Y de paso le damos un superpoder nuevo: llamar a cualquier API de internet."

"Para verlo, vamos a construir paso a paso un asistente útil y real: un conserje de viajes. Le dices un destino y te prepara un mini-informe: el tiempo que hará, un par de sitios imprescindibles, y qué está pasando allí ahora mismo. Y, como en la vida real, el informe no se da por bueno hasta que una persona lo aprueba."

Concepto · ¿cómo elige el agente una herramienta?

~5 min
🗣️ Decir

"Conectas tools al agente y n8n le manda al modelo una lista: nombre, descripción y parámetros de cada una. El modelo lee eso + la pregunta y decide. Tú NO escribes 'si pregunta el tiempo, usa get_weather'."

"Para el modelo, la descripción ES la herramienta. El código de detrás le da igual."

El botón ✨ y $fromAI()
🗣️ Decir

"La URL necesita la ciudad… pero no sé cuál pedirá el usuario. Que la rellene el modelo."

🖱️ Hacer

En URL escribe https://wttr.in/ → pasa el ratón por el final → icono ✨ → clic. Te queda así (puedes pegar esto directamente, en modo Expression):

URL del tool (con $fromAI)
https://wttr.in/{{ $fromAI('city', 'The destination city name in English, e.g. Lisbon or Paris', 'string') }}
🗣️ Decir

"$fromAI('city', …) es un hueco que rellena el modelo. La previsualización gris de abajo aún no resuelve; al ejecutar, el agente lee la pregunta, ve Lisboa y pone city=Lisbon → https://wttr.in/Lisbon."

Output · campo trip_brief (Expression)

Anillo 2 · + Wikipedia + Google Search

el agente elige~15 min

Con varias tools, lo interesante es cuál elige y por qué.

🖱️ Hacer
  • + Tool → Wikipedia (sin credenciales).
  • + Tool → SerpAPI → renómbralo Google Search (credencial gratis).
  • Reemplaza el System Message por este:
System Message completo (anillo 2)
You are a Trip Briefing assistant. The user names a destination (and sometimes dates) and you prepare a short, practical trip brief.

You have three tools:
- get_weather: live weather for a city. Use it whenever the user wants the weather or what to pack.
- Wikipedia: established background facts (history, main sights, geography) that rarely change. Use it for general "tell me about" knowledge.
- Google Search: current, time-sensitive info (events this week, opening hours, ticket prices, recent news). Use it when the answer depends on what is happening NOW.

Rules:
- Choose the RIGHT tool: stable facts -> Wikipedia; current/this-week info -> Google Search; weather -> get_weather.
- Do NOT invent opening hours, prices or dates. If it is time-sensitive, search.
- Final answer: a short brief of 4-6 bullets (weather, 2-3 things to see, 1 current tip).
🗣️ Decir

"Wikipedia y Google se parecen — las dos 'buscan'. ¿Cómo elige? Por la diferencia que marco: hechos que no cambian → Wikipedia; cosas de AHORA → Google. Las dos son nativas (descripción fija), así que las dirijo desde el System Message."

Pista: cuando no puedes tocar la descripción de una tool, la guías con el system message.

🖱️ Demo

Execute → agente → Logs: get_weather (tiempo) + Wikipedia (Belém, Castillo) + Google (eventos del finde). Cada tool para lo suyo, decidido por el agente.

Anillo 3 · La descripción ES la interfaz

VARIACIÓN EN VIVO 🌟~15 min

Demostrar la tesis rompiendo una descripción.

🖱️ Romper

En get_weather, sustituye la Description por algo vago:

Description "mala"
Checks the weather.

Execute con una pregunta ambigua:

Pregunta de prueba (cambia el request)
I'm going to Porto on Saturday — is it a good day to walk around?
🗣️ Decir

"Con descripción pobre el modelo duda: a veces no llama a la tool y se inventa el tiempo, a veces la llama mal. El nombre get_weather no basta."

🖱️ Arreglar

Restaura la descripción buena y vuelve a ejecutar la misma pregunta → ahora la llama bien.

Description "buena" (restaurar)
Get the current weather for a city. Input: the destination city name in English (for example "Lisbon"). Returns temperature, conditions, humidity and wind. Use this when the user asks about the weather or what to pack. Do NOT use it for opening hours, events, prices or general facts.
🗣️ Decir

"Cambié UN campo de texto y el comportamiento cambia entero."

"Escríbela como a un fichaje nuevo: qué hace, cuándo SÍ, cuándo NO, qué entra y sale. 3-4 frases."

"Provide extremely detailed descriptions. This is by far the most important factor in tool performance." Anthropic
(Opcional) Variación vía system message

"¿Forzar que SIEMPRE consulte el tiempo? No toco la descripción: añado una regla al system message."

Regla extra para el System Message
Always include current weather in the brief, even if the user does not ask for it.

Anillo 4 · Lectura vs escritura + aprobación

write~12 min
🗣️ Decir

"Todas nuestras tools son de SOLO LECTURA: buscan, no cambian nada. El día que una tool envíe un email o borre algo, un error sale caro."

Regla de oro: si una tool cambia algo en el mundo real, necesita aprobación humana antes de ejecutarse.

🖱️ Hacer
  1. Entre Agent y Output, añade Wait → Wait — Approval. Resume: On Webhook Call.
  2. En Options, Webhook Suffix (Expression):
Wait → Webhook Suffix
{{ $execution.id }}

Reconecta: Trip Briefing Agent → Wait — Approval → Output — Approved. En el Output añade un 2º campo:

Output · campo status (valor fijo)
approved

(activa Keep Only Set)

🖱️ Demo

Execute → se pausa en Wait. Clic en el nodo → copia la Test URL → ábrela → continúa a status: approved.

🗣️ Decir

El patrón: la IA prepara → el humano aprueba → el sistema ejecuta.

Resumen

~5 min
AnilloQué aprendimos
1Una tool HTTP llama a cualquier API; $fromAI() rellena los parámetros
2Con varias tools el agente elige; la descripción (o el system message) decide
3La descripción ES la interfaz: cámbiala y cambia el comportamiento
4Lectura vs escritura: las acciones que cambian el mundo van tras un gate de aprobación
🗣️ Tres ideas para llevarse
  1. El modelo elige la tool leyendo su descripción (qué hace + cuándo SÍ + cuándo NO).
  2. $fromAI() = el modelo rellena los argumentos en tiempo de ejecución.
  3. Read-only se usa libre; write necesita aprobación.
