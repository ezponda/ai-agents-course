1 · Puente desde Reflection

~4 min

Objetivo: enlazar con lo que ya vieron y llegar a la frase "ese nodo es el de hoy". Recordatorio, no clase nueva.

🗣️ Qué es el Reflection Pattern (recuérdalo así)

"Antes de meternos en el nodo de hoy, recordemos rápido la clase pasada, porque enlaza directamente. Vimos el Reflection Pattern: la idea de que el modelo no tiene que acertar a la primera. Igual que una persona, puede escribir un borrador, mirarlo con ojo crítico, ver qué falla y corregirlo, y repetir eso hasta que queda bien. En una frase: generar → criticar → corregir → repetir. Es un bucle de auto-mejora."

👀 El ejercicio que estábamos viendo

"El ejercicio era a propósito difícil para un LLM: escribir la descripción de un producto —unos auriculares Bluetooth— que cumpliera varias restricciones a la vez:"

  • exactamente 3 frases,
  • entre 25 y 30 palabras,
  • que contenga la palabra "sound" dos veces,
  • y que no use "music" ni "audio".

"Un modelo casi nunca clava todo eso de un tiro — por eso necesita el bucle de reflexión: lo intenta, comprueba, y vuelve a intentarlo."

Las dos versiones (esto es lo que enlaza con hoy)
🖱️ Hacer · abre 05 y enseña las dos mitades

"Lo montamos de dos maneras, una al lado de la otra. Mirad."

V1 (a mano): Generate → Validate → Parse → [ If Done? ] ──sí──▶ Output ▲ └──no──▶ vuelve a Generate V2 (agente): Input → [ AI Agent + Validator ] → Output (un solo nodo lo hace todo)
🗣️ Decir · V1 vs V2

V1: "Aquí el bucle lo cableamos nosotros: el modelo genera, otro paso valida si cumple las reglas, y este nodo If Done? decide — si cumple, sale; si no, vuelve atrás a reintentar. La condición de salida la pusimos a mano: 'para cuando cumpla todo o cuando llegue a 5 intentos'. Nosotros diseñamos el bucle."

V2: "Y aquí lo mismo… pero con un solo nodo: el AI Agent. Le damos el objetivo y una herramienta para validar, y él decide solo cuándo ha terminado: escribe, comprueba, corrige, y para cuando está bien. Donde en V1 había media docena de nodos, aquí hay uno."

🗣️ La frase-puente (remate hacia hoy)

"Ese nodo de V2 que lo hacía todo solo — el AI Agent — es justo el protagonista de hoy. Hasta ahora lo habéis usado como una cajita mágica; vamos a abrirla y entender qué tiene dentro y por qué funciona."

⚠️ Ojo

No te metas a explicar el Validator ni a ejecutar el bucle entero (eso fue la clase de Reflection). Aquí solo es el trampolín. Máximo 4 min.

2 · El nodo AI Agent (Ejemplo 3)

~18 min

Abrir la "cajita": sus partes, el system message, cómo piensa (el bucle), la memoria y los guardarraíles.

🖱️ Hacer

Abre 07 — AI Agent Chat (Ex3) (el nodo se llama AI Agent — Salary Chat). Es un asesor de salarios por chat.

🗣️ Presenta el ejemplo

"Para verlo usamos un ejemplo concreto: un asistente de salarios. Le preguntas por chat cosas tipo '¿cuánto cobra un data scientist en España?' y te responde buscando datos reales y haciendo cuentas. Vamos a ver cómo está hecho por dentro."

2.1 · ¿Qué hace exactamente el nodo? (ReAct + planning)
🗣️ Decir · qué hace por dentro

"Antes de mirar las piezas: ¿qué hace este nodo exactamente? Por dentro sigue un patrón llamado ReAct — de Reason + Act, razonar y actuar. En vez de soltar la respuesta de un tirón, el agente alterna pensar y hacer en un bucle: piensa qué necesita, actúa (llama a una herramienta), observa el resultado, y vuelve a pensar — hasta que tiene la respuesta."

ReAct = Reason + Act, en bucle: THINK "necesito el sueldo medio en España" ACT Google Search("data scientist salary Spain") OBSERVE ~35.000 € THINK "ahora el neto al 24%" ACT Calculator(35000 * 0.76 / 12) OBSERVE ~2.216 €/mes THINK "ya tengo la respuesta" FINAL responde al usuario
🗣️ Planning + cómo procesa

"El paso THINK es planificación: el agente se hace un plan antes de lanzarse (en Ex3 hay incluso una herramienta, Think, que lo hace explícito). Y cómo procesa: en cada vuelta el modelo recibe la pregunta + todo lo que las tools han devuelto hasta ese momento, y decide UNA cosa — ¿llamo a otra herramienta o ya respondo?. Cuando decide que ya puede, sale del bucle."

👀 Diferencias (tenlo a mano · de la teoría del curso)
Enfoque¿Quién decide los pasos?¿Las tools?¿Bucle?
LLM simple (Basic LLM Chain)tú (un prompt)sin toolsno
Workflowtú (pasos fijos)tú (al diseñar)no
Agentela IA (en runtime)la IA (en runtime)sí
🗣️ Regla práctica + un detalle técnico

"La regla de oro: si puedes escribir todos los pasos por adelantado → workflow; si la IA tiene que averiguarlos → agente. Y un detalle de n8n: un Basic LLM Chain devuelve su texto en el campo text; el AI Agent lo devuelve en output — porque no es una respuesta suelta, es el final de un bucle."

🔑 Clave (en una línea)

Agente = un LLM que, en bucle, DECIDE qué herramienta usar y cuándo parar. LLM normal = una pregunta, una respuesta.

2.2 · Anatomía: el nodo y sus piezas (líneas de puntos)
🗣️ Decir

"Fijaos en lo que cuelga del nodo por líneas de puntos. Una pista de n8n: las líneas sólidas son flujo de datos ('pasa al siguiente paso'); las de puntos son capacidades ('esto el agente PUEDE usarlo'). El AI Agent no trabaja solo: tiene cuatro piezas."

PiezaEn Ex3Cómo lo explico
Chat ModelOpenRouter"el cerebro — el LLM que razona"
MemorySimple Memory (10)"la memoria — recuerda los últimos mensajes de ESTA conversación"
ToolsCalculator · Google (SerpAPI) · Think"las manos — lo que puede hacer o consultar"
System Message(en Options)"el briefing — quién es y qué reglas sigue"
🗣️ Frase para fijar la idea

"Un agente = un LLM (cerebro) + memoria + herramientas (manos) + unas reglas. Eso es todo el misterio."

⚠️ Dato n8n

Desde v1.82 el AI Agent es siempre "Tools Agent" y necesita ≥1 tool conectada. Si veis menús de "tipo de agente" en tutoriales viejos, ya no existen.

2.3 · El System Message (ábrelo y léelo)
🖱️ Hacer

Abre el nodo → Options → System Message. Léelo en pantalla:

System Message de Ex3
You are a salary research assistant inside an n8n AI Agent workflow.
Be concise.
If the user asks about salaries, job markets, or company info, search Google first.
If the user asks for arithmetic, use the Calculator tool.
Use Think to plan multi-step research.
If the user asks you to remember something, store it in memory and confirm briefly.
If you don't know, say so.
🗣️ Decir · recorre el prompt

"Esto es el briefing del agente, y es donde está casi todo el trabajo. Mirad lo que le decimos: primero quién es ('asistente de investigación salarial') y que sea conciso. Luego cuándo usar cada herramienta: si preguntan por salarios o empresas, que busque en Google; si hay que calcular, que use la Calculadora; y que use Think para planificar antes de lanzarse. Le decimos también que guarde en memoria lo que le pidan recordar. Y una regla de oro contra invenciones: 'si no lo sabes, dilo'."

"Importante no confundir dos cosas: el System Message son las instrucciones fijas (siempre las mismas); el Prompt es el mensaje del usuario de cada turno — aquí viene del chat, con {{ $json.chatInput }}."

2.4 · El bucle en vivo: verlo en los Logs
🗣️ Decir

"Ese bucle ReAct que acabamos de ver en teoría, ahora vamos a verlo de verdad. Lanzo una pregunta que obligue al agente a encadenar varias herramientas y abrimos los Logs para ver, paso a paso, lo que decide."

🖱️ Hacer · verlo en los Logs

Doble clic en el AI Agent → pestaña Logs. Lanza por chat una pregunta que obligue a usar varias tools:

pregunta de prueba (chat)

👀 Mirar en los Logs · y narrarlo

"Mirad la secuencia de pasos que ha hecho él solo: primero Think — se hace un plan ('necesito buscar el sueldo y luego calcular el neto'); luego Google Search — busca y encuentra la cifra; luego Calculator — hace la cuenta del neto; y por fin redacta la respuesta. Cada paso con su entrada y su salida. Esto es el bucle en acción — y nadie le dijo el orden, lo decidió él."

🗣️ Gancho hacia tool calling

"Y aquí surge la pregunta del millón: ha elegido las herramientas correctas él solo… ¿cómo sabe cuál usar? Eso es lo que veremos en la segunda parte de la clase: el tool calling."

2.5 · La memoria (demo en vivo que engancha)
🗣️ Decir

"La otra pieza era la memoria. Vamos a verla funcionar. Le digo algo y luego le pregunto sin repetirlo, a ver si se acuerda."

🖱️ Hacer · en el mismo chat, sin cerrarlo
mensaje 1
Remember that I'm interested in jobs in Berlin.
mensaje 2
What city am I interested in?

→ responde "Berlin". Ahora abre un chat nuevo y repite el mensaje 2 → ya no lo sabe.

🗣️ Decir

"¿Veis? En la misma conversación se acuerda; en una ventana nueva, empieza de cero. La memoria va por sesión (un sessionId): misma sesión = misma conversación. Y un matiz importante para más adelante: memoria ≠ conocimiento. La memoria guarda lo que se ha dicho en el chat; no son tus documentos. Darle acceso a tus documentos es otra cosa (RAG), que veremos en su momento."

3 · Tool calling: ¿cómo elige una herramienta?

~6 min

Responder al gancho del bloque anterior: el agente elige leyendo la descripción de cada tool.

🗣️ Qué es una descripción de tool

"En el bloque anterior el agente eligió las herramientas él solo. ¿Cómo sabe cuál usar? Por la descripción de cada una. La descripción es el texto que le dice al modelo qué hace la tool y, sobre todo, cuándo usarla. Es el manual de instrucciones que ve el modelo — para él, la herramienta es su descripción; lo que haya por dentro le da igual."

🗣️ Cómo llega al modelo (la duda típica)

"¿Y eso dónde se mete, en el prompt? Buena pregunta. Cuando conectas tools al agente, n8n recoge de cada una su nombre + descripción + qué parámetros recibe y se los pasa al modelo como una lista estructurada de 'herramientas disponibles' dentro de la propia llamada a la API — no se pega como texto dentro de tu system prompt. No van 'arriba ni abajo' del prompt: van en un campo aparte. El modelo está entrenado para recibir esa lista y, en vez de contestar texto, responder 'llamo a la tool X con estos datos'. n8n ejecuta la tool y le devuelve el resultado."

Tu llamada al modelo lleva DOS cosas separadas: • messages: system message + el mensaje del usuario • tools: [ {name, description, parámetros}, ... ] ← aquí van las descripciones El modelo responde: "usar get_weather con city='Lisbon'" → n8n la ejecuta → resultado → vuelve al modelo
👀 De dónde sale cada descripción
Tipo de toolDescripción¿La escribes tú?
Nativa (Wikipedia, Calculator)fija, viene de fábricaNo
Custom (HTTP Request, Code, AI Agent Tool)un campo en la toolSí — y ahí está el 80% del éxito

"Para las nativas no puedes tocar la descripción → si quieres orientar la elección, lo haces desde el System Message ('usa Google para info actual'). Para las custom, la descripción la escribes tú."

🔑 Clave (en una línea)

La descripción es la interfaz que ve el modelo. Buena descripción → elige bien; vaga → falla. Es el factor nº 1.

🗣️ Regla para escribirlas

"La regla de oro: di cuándo SÍ y cuándo NO usar la tool, no solo qué hace. 'Busca en Wikipedia hechos sobre personas, lugares o eventos; NO la uses para noticias recientes.' 3-4 frases, como si se lo explicaras a alguien que entra nuevo."

3 · Build en vivo: Trip Briefing (por anillos)

~20 min

Construir un agente que prepara un mini-informe de viaje, añadiendo capacidades anillo a anillo. (Workflow: 09_tool_calling_trip_briefing.)

Antes del Anillo 1 · ¿Qué es una API? (la máquina de Coca-Colas)
🗣️ Decir · la analogía

"Una de las tools llama a una API, así que dos minutos de qué es eso. Pensad en una máquina de Coca-Colas: haces una petición (metes la moneda y pulsas un botón) y la máquina te da una respuesta (la lata). Una API es igual pero por internet: tu workflow manda una petición a una URL y el servidor le devuelve datos (normalmente en JSON)."

👀 Las piezas de una petición
PiezaMáquina de Coca-ColasAPI
URL / endpointla máquina concretala dirección a la que llamas (https://wttr.in/Lisbon)
Métodoqué acción hacesGET = pedir/leer datos · POST = enviar/crear algo
Parámetrosqué botón pulsasafinan la petición (?format=j1, la ciudad…)
Respuestala lata que caelos datos devueltos (JSON)

"Hoy solo hacemos GET (leer): pedir el tiempo. Un POST sería crear un pedido o enviar un mensaje — eso ya cambia algo, y lo que cambia el mundo necesita más cuidado (aprobación)."

🗣️ La API que usamos: wttr.in

"Usamos wttr.in: API del tiempo, gratis y sin clave. La URL se construye así: https://wttr.in/CIUDAD + ?format=j1 para que devuelva JSON. Ábrela en el navegador antes de clase para enseñar la 'lata' que cae."

pruébalo en el navegador (esto es un GET)
https://wttr.in/Lisbon?format=j1

Web de la API: github.com/chubin/wttr.in · formatos: output formats

👀 ¿Y el format=j1? (por si lo preguntan)

wttr.in puede devolver el tiempo de varias formas. Sin format devuelve un informe en texto/ASCII (bonito para humanos, ruidoso para el modelo) — por eso "funciona igual". Con format=j1 devuelve JSON estructurado (limpio y fácil de leer para el agente). Otros: format=3 (una línea), format=4. Para un agente, mejor j1."

ábrelas una a una y compara lo que devuelve
https://wttr.in/Lisbon
https://wttr.in/Lisbon?format=3
https://wttr.in/Lisbon?format=j1
Anillo 1 · get_weather (HTTP Request Tool + $fromAI)
🗣️ Decir

"Empezamos con una tool potente: una que llama a una API de internet cualquiera. Le damos al agente el tiempo en vivo. Lo nuevo aquí es el HTTP Request Tool — deja al agente llamar a una URL — y $fromAI, que es cómo el modelo rellena los parámetros de la llamada."

🖱️ Cómo encontrar / abrir el nodo
  • Si ya existe: clic en el nodo get_weather (cuelga del agente por la línea de puntos Tool) para abrirlo.
  • Para añadirlo desde cero: en el AI Agent, clic en el + del conector Tool → busca "HTTP Request" → renómbralo get_weather.

👀 Mirar

La previsualización gris de la URL: aún no resuelve. Al ejecutar, el modelo pone city=Lisbon → https://wttr.in/Lisbon. Eso es $fromAI: el modelo rellena el hueco.

🗣️ Duda típica: ¿$fromAI llama otra vez al modelo? (anticípala)

"No es una llamada extra ni magia de n8n. Cuando el agente decide usar get_weather, ya está hablando con vuestro modelo de OpenRouter — y en esa misma respuesta el modelo devuelve los argumentos de la herramienta, incluido city='Lisbon'. $fromAI solo le dice a n8n: 'este hueco lo rellena la IA'."

Resumen: el valor lo decide el modelo (OpenRouter) como parte de su tool call; n8n coge ese valor, lo mete en la URL y hace la petición HTTP. Mismo turno, sin coste de una llamada aparte.

⚠️ Errores posibles
  • El $fromAI no se rellena → el campo URL debe estar en modo Expression (icono fx), no Fixed.
  • 404 / "Unknown location" → ciudad mal escrita o con espacios: "New York" → New+York (o usa "Paris", "Lisbon").
  • Respuesta enorme o lenta → deja Optimize Response ON; si diera guerra, pruébalo OFF.
  • "Bad request" en el agente → casi nunca es esta tool: mira si hay otra tool en ⚠️ (credencial) o el bug de Think.
Mini-ejercicio · SIN $fromAI: un Code Tool que recibe la ciudad como query
⚠️ Lo que comprobamos en clase (importante)

Probamos a poner {city} en la URL del HTTP Request Tool y NO funciona: n8n avisa "No parameters are set up to be filled by AI. Click on the ✨ button…" y manda {city} literal (devuelve un sitio por defecto, no Lisboa). En el HTTP Tool, la ÚNICA forma de que el agente rellene un valor es $fromAI (botón ✨). Así que para hacerlo sin $fromAI y que el agente igual pase la ciudad → usamos un Code Tool, donde el agente te pasa su texto en query.

👀 Los 3 estados (de más a menos ayuda al modelo)
EstadoCómo recibe la ciudad¿Quién decide la ciudad?
1. $fromAI descrito (HTTP Tool)$fromAI('city', 'The destination city in English')el modelo, guiado por la descripción
2. Sin $fromAI (Code Tool)la variable query en el código + reglas en el System Messageel modelo, guiado por tu prompt (más frágil)
3. Sin huecohttps://wttr.in/Lisbon (fija)nadie — fija; deja de ser agéntico

Hacemos el estado 2 en vivo (el 1 ya está en el Anillo 1; el 3 es el ⚠️ de abajo).

🖱️ Hacer · qué cambiar (paso a paso)

Cambio 1 — usa un Code Tool en vez del HTTP Request Tool. Borra get_weather (el HTTP) y añade un Code Tool (JavaScript), llámalo igual get_weather. El agente te pasa su texto en la variable query — sin $fromAI. Pega este código:

Code Tool · JavaScript
const city = String(query).trim();
const url = `https://wttr.in/${encodeURIComponent(city)}?format=j1`;
const res = await this.helpers.httpRequest({ url, json: false });
return String(res);

Cambio 2 — el System Message del agente (el truco). El parseo lo hace el agente: que extraiga la ciudad, la traduzca a inglés y pase SOLO eso como input del tool. Reemplaza el System Message por este:

System Message completo (el agente parsea y pasa la ciudad)
You are a travel weather assistant.

When the user asks about the weather, call the get_weather tool. The tool needs the city prepared in a very specific way, so YOU must prepare it BEFORE calling:
- Extract the city the user means.
- Translate it to its English name (e.g. "Lisboa" -> "Lisbon", "Londres" -> "London").
- Pass ONLY that city as the tool input — just the city name, no extra words, no "weather in", no punctuation, no country.

Then read the tool's result and answer the user in plain language.

Cambio 3 — la Description del Code Tool. Dile al agente para qué es y qué pasarle:

Description del Code Tool
Returns the current weather for a city. Input: a single city name in English.

Probar: pregunta "¿qué tiempo hace en Lisboa?" y mira en los Logs qué pasó el agente como query → debería ser Lisbon (no "Lisboa", no "weather in Lisbon").

⚠️ Ojo

El Code Tool necesita poder hacer la petición (this.helpers.httpRequest); en algunas versiones de n8n hay que probarlo. Si fallara, plan B: volver al HTTP Request Tool con $fromAI (✨).

👀 Qué cambia respecto al original

Cómo recibe la ciudad$fromAI('city', '…') en la URL (botón ✨)la variable query en el código
Quién guía el formatola descripción del parámetroel System Message (extraer / traducir / pasar solo la ciudad)
Quién hace la llamadan8n (el nodo HTTP)tu código (httpRequest)
🗣️ La moraleja

"En el HTTP Tool el agente solo rellena vía $fromAI (✨). Para hacerlo sin $fromAI, el Code Tool: el agente pasa la ciudad por query y el formateo lo defines TÚ en el system prompt."

"Con $fromAI declaras el parámetro y su descripción guía al modelo; con el Code Tool, el 'contrato' de qué pasar lo pones en el prompt — más control, más frágil. Mismo resultado, distinto reparto del trabajo."

⚠️ Estado 3 · ciudad fija (sin agente)

Si en vez de query pones la ciudad fija en el código (o usas el HTTP Tool con la URL https://wttr.in/Lisbon sin $fromAI), el agente ya no puede cambiar la ciudad — siempre devuelve Lisboa. Deja de ser agéntico.

💡 Más ejercicios didácticos (elige)
  • Romper $fromAI: en el tool del agente, fija la ciudad en la URL (quita el $fromAI) y pregunta por OTRA ciudad → siempre devuelve la fija. Demuestra para qué sirve.
  • Otro idioma: pregunta "¿qué tiempo hace en Lisboa?" → el modelo igualmente rellena city=Lisbon. Enseña que $fromAI no es copia-pega: el modelo interpreta.
  • Cambiar el format: pon format=3 y vuelve a preguntar → el agente recibe una línea en vez del JSON; comenta cómo la forma de la respuesta afecta a lo que puede hacer.

🖱️ Hacer · demostrar la tesis

En get_weather, cambia la descripción por una vaga y re-ejecuta:

descripción "mala"
Checks the text.

El agente duda o no la llama. Restaura la buena (la del Anillo 1) → vuelve a funcionar.

🗣️ Decir

"Cambié UN campo de texto y el comportamiento cambia entero."

"El nombre get_weather no basta: el modelo solo la conoce por su descripción. Por eso es el factor nº 1."

Anillo 4 · Lectura vs escritura + aprobación
🗣️ Decir

"Todas estas tools son de solo lectura: buscan, no cambian nada. El día que una tool envíe un email o borre algo, un error sale caro. Regla de oro: si una tool cambia algo en el mundo real, necesita aprobación humana antes de ejecutarse. Aquí lo vemos con un nodo Wait que pausa el flujo hasta que apruebas el informe."

👀 Mirar

El flujo se pausa en Wait — Approval; abres la Test URL, apruebas, y solo entonces sale el informe final.

4 · Cierre + Q&A

~5 min
🗣️ Recapitular

"Resumiendo la clase de hoy: un AI Agent es un LLM con memoria, herramientas y reglas que trabaja en bucle (ReAct) decidiendo qué hacer. Usa las herramientas leyendo su descripción — el factor nº 1 — y rellena sus parámetros con $fromAI. Y lo que cambia el mundo (escribir, enviar) va siempre detrás de una aprobación."

🔑 Tres ideas para llevarse
  • Agente = LLM + bucle + tools (decide y repite).
  • La descripción es la interfaz: decide qué tool usa.
  • Read-only se usa libre; write necesita aprobación.
