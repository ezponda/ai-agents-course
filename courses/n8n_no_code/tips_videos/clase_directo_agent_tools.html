1 · ¿Workflow o agente?

8 minREPASO + ENCUADRE

🗣️ El gancho

"En los vídeos anteriores vosotros decidíais cada paso: encadenar prompts, un Switch, un If. Hoy entramos de lleno en el nodo que cambia las reglas. La pregunta de toda la clase: ¿qué decisión dejo de cablear y se la doy al modelo?"

🗣️ Las dos definiciones (Anthropic)
  • Workflow = "LLMs y herramientas orquestados por rutas de código predefinidas." → tú escribes el guion.
  • Agente = "el LLM dirige dinámicamente sus procesos y el uso de tools." → tú das un objetivo.

"Un agente = LLM + bucle + tools."

⚠️ Riesgo en directo

V1 puede no converger y agotar las 5 iteraciones (validador LLM). Perfecto para el argumento, pero no te juegues el "wow" a V1 — apóyate en V2.

3 · El bucle delegado: V2, el agente decide

12 min

Misma tarea; ahora el agente gestiona el bucle solo.

🖱️ Hacer
  1. Ejecuta Run: V2 Agent.
  2. Clica Reflection Agent (V2) → tool calls: escribe → llama al Validator → lee issues → reescribe → revalida hasta all_pass.
  3. Compara con V1 lado a lado.
🗣️ Tres remates
  1. "El agente decide cuándo ha terminado" (cuando ya no llama tools). Es el agent loop: reunir contexto → actuar → verificar → repetir.
  2. El Validator es código, no un LLM. Jerarquía de verificación (Anthropic):
    🥇 reglas/tests · 🥈 visual · 🥉 LLM-as-judge (el menos fiable).
  3. "Los agentes no son magia: automatizan el mismo bucle que montaste a mano en V1."
"The best form of feedback is providing clearly defined rules… LLM-as-judge is generally not a very robust method."— Anthropic, Building agents with the Claude Agent SDK

4 · 🌟 Tool calling y descripciones

20 minNÚCLEO NUEVO

Lo que NO está en ningún vídeo. El corazón de la clase.

🗣️ La pregunta y la respuesta

"El agente 'llama al Validator'… ¿pero cómo sabe cuándo y cómo usarlo?"

"Lee su DESCRIPCIÓN. Para el modelo, la descripción ES la herramienta."

"Provide extremely detailed descriptions. This is by far the most important factor in tool performance."— Anthropic, docs de tool use

🔬 Capa 1 — "La descripción es la interfaz" imprescindible

🖱️ Romper y arreglar
  1. Abre el tool Validator; lee su descripción actual (buena).
  2. Rómpela → Checks the text. → ejecuta V2 → el agente duda / cuenta él mismo / se degrada.
  3. Restáurala → vuelve a funcionar.
🗣️ Frases que se quedan
  • "El nombre Validator no basta. El modelo solo conoce la tool por su descripción."
  • "Descríbela como a un fichaje nuevo" · "como un buen docstring para un junior."
  • ACI = HCI: dedica a la interfaz agente-ordenador tanto esfuerzo como a una de usuario.
  • Heurística: 3–4 frases, y di cuándo SÍ y cuándo NO usarla.

Ejemplo de pizarra (Anthropic):

❌ Mala:  "Gets the stock price for a ticker."

✅ Buena: "Retrieves the current stock price for a given ticker symbol.
        The ticker must be a valid symbol on a major US exchange (NYSE/NASDAQ).
        Returns the latest trade price in USD. Use it when the user asks for the
        current/most recent price of a stock. It will not return any other info."
🗣️ System message ≠ descripción

El system message = las reglas ("no cuentes tú, usa el Validator"). La descripción = qué hace y cómo llamarla. Reglas ≠ manual de la herramienta.

🔬 Capa 2 — Multi-tool: cómo ELIGE el núcleo

🖱️ Partir el Validator en dos
  1. Crea LengthChecker (frases + palabras) y KeywordChecker ("sound" + prohibidas). (código en Anexos)
  2. Descripciones claras y distintas → el agente elige la correcta (a veces las dos).
  3. El "aha": hazlas solapar ("checks the description") → elige mal / redundante → afínalas → arreglado.
🗣️ Remates
  • "Con varias tools, la descripción es lo que las desambigua."
  • Test del límite: "si no puedes decir en una frase por qué elegiría la tool A en vez de la B, no está lista."
  • Menos tools, mejores: "más herramientas no dan mejores resultados" (OpenAI: <20 por turno).

🔬 Capa 3 — ¿Con qué argumento la llama? $fromAI() opcional · puente

🗣️ Decir

"Tool calling = elegir la tool + rellenar sus argumentos. ¿Quién decide qué texto mandar? El modelo, con $fromAI(): un hueco que rellena."

Solo en tools del AI Agent (no en el Code tool). Puente a la próxima clase: HTTP Request Tool + $fromAI() = llamar a cualquier API. Luego: AI Agent Tool (multi-agente) y MCP.

5 · Cuándo NO usar agente + cierre

5 min
🗣️ Honestidad = credibilidad
  • Anthropic: "workflows = previsibilidad para tareas bien definidas; agentes = flexibilidad y decisión del modelo a escala."
  • Caso real: en la comunidad n8n montaron la misma clasificación de facturas de las dos formas — "el agente perdió": enrutaba distinto en cada ejecución y "te dice tan tranquilo que está 90% seguro de algo que falló."
  • Clasificar es determinista: no metas un agente donde un Switch sobra.
  • Regla n8n: "construye el workflow primero; añade IA solo donde haya ambigüedad."

📋 Anexos — copy-paste

A1 · Descripción "mala" (romper)

Checks the text.

A2 · Descripción "buena" (restaurar)

Validates a product description against the constraints. Input: ONLY the description text (no extra words like "Here is my draft:"). Returns JSON with sentences (count), words (count), sound_count, has_forbidden, all_pass (true/false) and issues (list of specific problems to fix). Use this after writing or revising a draft, before returning the final answer.

A3 · Tool LengthChecker

Description:

Counts the number of sentences and words in a product description. Input: ONLY the description text. Returns JSON: { sentences, words, length_ok }. Use this to check the length/structure constraints (exactly 3 sentences, 25–30 words). Do NOT use it for keyword checks.

JavaScript:

const item = $input.first().json;
let text = '';
for (const k of ['query','input','chatInput','action_input']) { if (item[k]) { text = String(item[k]).trim(); break; } }
if (!text) { for (const v of Object.values(item)) { if (typeof v === 'string' && v.trim()) { text = v.trim(); break; } } }

const sentences = text.split(/[.!?]+/).filter(s => s.trim()).length;
const words = text.split(/\s+/).filter(w => w).length;
const length_ok = sentences === 3 && words >= 25 && words <= 30;

return [{ json: { input_received: text.slice(0,200), sentences, words, length_ok } }];

A4 · Tool KeywordChecker

Description:

Checks keyword rules in a product description. Input: ONLY the description text. Returns JSON: { sound_count, has_forbidden, keywords_ok }. Use this to verify the word "sound" appears exactly twice and that "music"/"audio" are absent. Do NOT use it to count sentences or total words.

JavaScript:

const item = $input.first().json;
let text = '';
for (const k of ['query','input','chatInput','action_input']) { if (item[k]) { text = String(item[k]).trim(); break; } }
if (!text) { for (const v of Object.values(item)) { if (typeof v === 'string' && v.trim()) { text = v.trim(); break; } } }

const sound_count = (text.toLowerCase().match(/\bsound\b/g) || []).length;
const has_forbidden = /\b(music|audio)\b/i.test(text);
const keywords_ok = sound_count === 2 && !has_forbidden;

return [{ json: { input_received: text.slice(0,200), sound_count, has_forbidden, keywords_ok } }];
