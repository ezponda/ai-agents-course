1. Veredicto global

El curso está por encima de la media del género (media ~7.4/10). Dos fortalezas raras y valiosas:

  • El arco raw-loop → framework es el consenso del sector. DeepLearning.AI (AI Agents in LangGraph) empieza con "Build an Agent from Scratch"; Hugging Face Agents Course Unit 1 construye un agente dummy a mano; Ed Donner dedica su primera semana a fundamentos sin framework; la guía de OpenAI define el "run" como un while loop. El curso ya está donde hay que estar.
  • La escalera de memoria 06 → 14 → 23 → 28 (gestionar la lista → message_history → thread → usuario cross-thread) y el anclaje constante "esto es lo que hacías a mano en 0.X" (patrón Rosetta) son de lo mejor diseñado del curso.

Tres debilidades sistémicas:

DebilidadEvidencia
Ejercicios no verificables46+ ejercicios en dropdowns, 0 con assert / solución oculta / expected output. El Bloque 2 es el peor: 1 de 9 notebooks tiene ejercicios. Justo el ADN que el curso n8n sí tiene ("Done when:", "Expected:", escenarios adversariales).
El Bloque 2 es un callejón sin salidaNada de LangGraph (checkpointer, interrupt, stream) se usa en los capstones ni en el deploy. 9 notebooks sin desembocadura narrativa.
"Producción" que no produce31 es el notebook más flojo (5.5): nada ejecutable, seguridad reducida a un {note}. Sin auth, sin Docker, sin deploy guiado de extremo a extremo.

7. Propuesta detallada: capítulos 07 y 08

La numeración del Bloque 0 (00–06) deja libres 07 y 08. El hilo narrativo que sugieren todas las fuentes: el bucle es el 5% (Ball); el 95% restante es ingeniería del contexto y de las tools — y cada agente comercial resuelve las mismas capas con decisiones distintas y documentadas.

07_build_a_coding_agent

  • Tools de Ball/Huntley sobre el bucle de 0.3: read_file (offset/límite de líneas, como Claude Code, máx. 2000), list_files, edit_file (string-replace con validación de unicidad), bash (timeout + truncado de output).
  • Inyección de entorno: cwd + git status + árbol de ficheros en el system prompt (lo que hacen Claude Code y Codex con CLAUDE.md/AGENTS.md).
  • Permisos: interceptar tool calls con input() + allowlist de prefijos — el modelo mental de Codex: sandbox (qué puede) × approval policy (cuándo pregunta).
  • Clímax: "escribe la función, corre pytest, itera hasta verde" — el bucle de verificación (Crawshaw: "agents are feedback-driven LLMs").
  • Cierre comparativo: Claude Code = LLM-search (grep/glob, no RAG) + string-replace + permisos · Codex = patch V4A entrenado + sandbox OS (Seatbelt/Landlock) · Aider = repo-map (tree-sitter + PageRank) + formatos por modelo, con el dato de benchmark: udiff subió el éxito de GPT-4 Turbo del 20% al 61%.

08_context_engineering

  • Context rot: por qué el historial degrada el rendimiento (attention budget, Anthropic).
  • Compaction manual: al superar N tokens, resumir mensajes viejos con una llamada aparte — demostrable con límite artificial bajo (lo que hace el auto-compact de Claude Code).
  • Subagente: tool task(prompt) que lanza un segundo bucle con contexto limpio y devuelve solo el resumen (~30 líneas reutilizando su propia clase). Restricción real de Claude Code: máximo una rama — "debuggability >>> multi-agent mishmash" (MinusX).
  • TodoWrite: una tool que guarda/muestra la lista de tareas del propio modelo (~20 líneas) — combate el context drift; plan mode como contraste ReAct vs plan-then-execute.
  • Teoría con números: prompt caching (por qué reenviar todo el historial es viable; por qué la compaction interactúa con la cache) y model routing (>50% de las llamadas de Claude Code van a Haiku; $0,11 y 40s para una tarea trivial).

No construible en notebook (contar, no construir): sandboxing OS-level, repo-map completo, parser robusto de patches (sin matching flexible, 9× más errores según Aider), TUI, hooks.

8. ADN didáctico del curso n8n a portar

  1. Fallar primero, arreglar con nombre y apellidos (iteración V1→V2→V3 con "what you'll see go wrong" predicho — el mejor recurso del curso n8n, en project_2 y appendix_d).
  2. Test scenarios numerados con "Expected:" + al menos un caso adversarial (injection, dato inexistente, carrera de concurrencia).
  3. Challenges con "Done when:" verificable — incluidos retos de romper, no solo de construir (el "Break It — the Double-Booking Race" de project_7).
  4. Prompts anotados línea→patrón (tabla "Why it works").
  5. Seed data diseñado para la demo (huecos libres a propósito, fila cancelled para el soft-delete).
  6. Tablas síntoma→causa→arreglo + sesión de debugging narrada.
  7. Cada proyecto termina tocando el mundo (URL pública, schedule, "have a friend use it").

El curso n8n es didácticamente más fuerte en el ciclo de verificación por el alumno; es el ADN que un curso técnico suele omitir por asumir que "el técnico ya sabe probar". No transfieren: mecánica de UI, justificaciones sin-credenciales, vocabulario repetido, cap. de setup con capturas.
